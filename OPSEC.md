# PlausiDen-Inject — Operational Security

`plausiden-inject` is the platform-adapter layer of the PlausiDen
Protection Suite. It takes artifacts generated by `plausiden-engine`
and writes them into real OS data stores: Firefox's `places.sqlite`,
Chrome's `History`, Android content providers, macOS user defaults,
Windows registry, and so on.

This is the **highest-risk repo in the suite** because it holds the
only code that physically mutates the user's device outside a
sandbox. A bug in Engine produces a bad artifact in memory; a bug in
Inject corrupts the user's actual data. Read this document before
running Inject on a machine whose state you care about.

Authored: Claude 3, 2026-04-17.

---

## 1. Who this guidance is for

- **Integrators** wiring Inject into Desktop (Tier 2), Android (Tier
  1), or USB (Tier 4) hosts.
- **Operators** running Desktop or Android with Inject active.
- **Maintainers** of the per-platform adapter crates
  (`inject-linux`, `inject-macos`, `inject-windows`,
  `inject-android`, `inject-ios`).

Not the target: casual end users of a finished product (Desktop
app, Android app) — they read the host's OPSEC.md, not this one.

---

## 2. Threat models covered

- **Partial-data forensic triage.** Injection populates the same
  stores a forensic examiner opens first (places.sqlite, cookies.db,
  notification caches). Injected entries must be indistinguishable
  from organic data or the tool advertises itself.
- **Cross-artifact correlation.** Examiners cross-reference cookies
  against history, downloads against filesystem, notification
  senders against contacts. Inject must preserve these invariants
  when writing (cookies get domains that appear in history, downloads
  get paths under the user's actual Downloads folder, etc.).
- **Reversible accidents.** Every write is bookkept in a transaction
  so a buggy run can be rolled back without leaving partial state.
  `inject-core::transaction` is the primitive; every platform
  adapter must use it.

---

## 3. Threat models NOT covered

- **Pre-existing data corruption.** If places.sqlite is already
  corrupt when Inject starts, Inject may observe inconsistent
  state. It does not repair damage. Run a host-side integrity check
  before enabling injection on a long-lived profile.
- **Concurrent browser activity.** Firefox locks places.sqlite while
  running. Inject's Firefox adapter skips writes when it detects the
  lock rather than forcing them. This is CORRECT — never wrestle
  the browser for its DB.
- **Kernel-level forensic tools** (Cellebrite, Magnet AXIOM). These
  bypass the host OS's file API and read raw blocks. Inject's
  writes produce on-disk blocks that look organic; but a NAND-level
  examiner can spot freshly-allocated cells, so for timing-sensitive
  threats, pre-seed Inject days before the seizure window.
- **Antivirus / EDR interference.** Some EDR agents flag processes
  that write to `places.sqlite` from outside Firefox. Inject's
  writes pattern is legitimate-looking, but EDR policy may still
  block. The audit surface is the Desktop app's self-check
  (planned, task #33) + Sentinel IPC status.

## 4. Per-platform operational considerations

### 4.1 `inject-linux`

**Firefox (`places.sqlite` via rusqlite):** writes `moz_places` +
`moz_historyvisits`. Respects Firefox's SQLite lock — blocks if the
browser is running; returns a specific error the host surfaces to
the user.

**Chrome (`History` SQLite DB):** similar model; respects the
browser's lock. DANGER: Chrome SYNCS history to cloud if Sync is
enabled. Injected entries will sync. The Desktop app should warn
the user if Sync is on before first Inject run.

### 4.2 `inject-android`

**ContentProvider writes:** the Android adapter uses the standard
ContentResolver API to write to Browser, Contacts, Calendar, SMS
providers. Requires permission grants at install time (declared in
the host app's manifest).

**Notification injection:** NotificationManager API path for
unrooted devices (v1.1 §5.1). Also supports direct NotificationLog
content-provider writes on rooted devices as a fallback.

**Call/SMS injection:** disabled by default. Android's Permission
Controller will flag any app requesting `READ_CALL_LOG` /
`READ_SMS`; enabling these permissions draws attention. The host
should only request them when the specific use case requires them
and must document the tradeoff in its own OPSEC.md.

### 4.3 `inject-macos` (scaffold)

Writes to Safari history, Mail attachments folder, Notes SQLite
DB. Blocked by System Integrity Protection (SIP) on many paths;
requires user-granted Full Disk Access.

### 4.4 `inject-windows` (scaffold)

Writes to Edge / Chrome History DBs, Outlook PST, Registry run
keys (carefully), Jump Lists. Windows Defender's controlled-folder-
access feature blocks writes to many user folders; host must
request allowlist before injection.

### 4.5 `inject-ios` (scaffold)

iOS sandboxing makes most injection paths inaccessible without
jailbreak. The adapter scope is limited to Shared Containers and
Photos metadata. Mostly defers to PlausiDenOS (Tier 3) for iOS-
class threat models.

## 5. Transaction discipline

Every Inject run opens a transaction via `inject-core::transaction`.
Writes are staged, validated (no duplicate primary keys, no
out-of-range timestamps, no NULL in required fields), then
committed atomically. If any validation fails, the entire batch
rolls back.

**Host responsibility:** do not bypass the transaction. Direct
calls into the adapter's internal write functions (if exposed)
will NOT receive rollback protection.

**Rollback is best-effort at the DB level.** SQLite's journal
mechanism handles it. For file-system writes (macOS attachments,
Android ContentProviders that don't support transactions), the
adapter simulates rollback via a delete-the-written-files pass.
A partial rollback is possible if the process is killed mid-
commit — the audit-logging in `inject-core::attribution_scrubber`
records enough state to manually reverse a partial batch.

---

## 6. Attribution scrubbing

`inject-core::attribution_scrubber` removes markers that would
identify injected entries as coming from Inject (specific user-
agent strings in HTTP caches, specific process names in
write-ahead logs, specific attribution fields in CMS data).

Scrubbing is AFTER validation — if validation fails, scrubbing
doesn't run, and the original markers never reach the store.

The scrubber is opt-out per adapter; disable only with documented
rationale in the host's configuration.

## 7. Known failure modes

- **SQLite lock contention.** If the browser opens between Inject
  checking the lock and Inject writing, the write fails with
  SQLITE_BUSY. The host retries up to 3 times with exponential
  backoff, then surfaces the error to the user.
- **Schema migration mid-run.** Firefox occasionally bumps its
  places.sqlite schema. If Inject runs on the old schema and
  Firefox migrates mid-run, foreign-key constraints may fail.
  Defense: Inject checks `user_version` PRAGMA first and aborts
  if the schema is newer than it expects.
- **Filesystem permissions.** Non-root Linux users cannot write
  to `/var/log` or `/etc`. Adapters that target those paths
  return permission errors instead of attempting with sudo.
- **ContentProvider permission revocation.** If the user revokes
  a granted permission mid-run on Android, subsequent writes
  throw SecurityException. Host surfaces via self-check.
